# Фильтр по периоду — дизайн

**Дата:** 2026-05-14
**Автор:** Соня (по запросу Вадима)
**Статус:** одобрен, готов к реализации

## Цель

Добавить в дашборд фильтр по временному диапазону. Сейчас все блоки всегда показывают полную историю (8 месяцев на момент проектирования). С ростом объёма данных это станет неудобно — графики перегружены, KPI «всего за всё время» теряют управленческий смысл.

## Принятые решения

| Вопрос | Решение |
|---|---|
| Scope фильтра | На все блоки, **кроме** карточек запасов |
| Тип контрола | Гибрид: пресеты + раскрытие кастома |
| Набор пресетов | Весь / Последний полный месяц / 3 / 6 / 12 месяцев / Кастом |
| Гранулярность | Авто на основе диапазона + ручное переопределение |
| Место в UI | Сверху сайдбара, над артикулами |
| Дефолт | Последние 3 месяца |

## Архитектура и поток данных

```
Пользователь выбирает пресет (или вводит кастом)
        ↓
_resolve_period(preset, custom_from, custom_to, analytics_max_period) → (start, end)
        ↓
_auto_granularity(start, end) → 'Неделя' | 'Месяц'
        ↓
sales[date.between(start, end)] → sales_period
analytics[period_start.between(start_month, end_month)] → analytics_period
        ↓
KPI, инсайты, графики, упущенная выручка, остаток+приходы, % отмен, таблица
    → работают на sales_period / analytics_period

Карточки запасов
    → работают на full analytics (last_month_an = max period_start от всей истории)
```

**Зависят от фильтра:**
- 4 KPI наверху
- Аналитические наблюдения (`build_insights`)
- Динамика продаж
- Упущенная выручка
- Остаток + Приходы
- % отмен по артикулам
- Сводная таблица

**Не зависят** (всегда полные данные):
- Анализ запасов (сводка «Заморожено в товаре» и карточки SKU)

## Семантика пресетов

Все расчёты идут от `analytics['period_start'].max()`, а не от `today` —
чтобы окно не «проскакивало» данные, которых ещё нет.

| Пресет | Логика |
|---|---|
| Весь период | `start=min, end=max` всех данных |
| Последний полный месяц | период последнего файла `analytics_report` |
| Последние 3 месяца | 3 последних месяца аналитики |
| Последние 6 месяцев | 6 последних месяцев |
| Последние 12 месяцев | 12 последних (если меньше — берётся что есть) |
| Кастом | `st.date_input` × 2 |

## Авто-гранулярность

| Диапазон | Авто |
|---|---|
| ≤ 31 день | Неделя |
| > 31 день | Месяц |

Пользователь может вручную переключить radio. При смене пресета ручной выбор
сбрасывается (упрощает поведение, не запутывает).

## Дельты KPI

Считаются как «выбранный период vs предыдущий период такой же длины».

- Apr 2026 (1 мес) → дельта vs Mar 2026
- Feb–Apr 2026 (3 мес) → дельта vs Nov 2025–Jan 2026

Если предыдущего окна нет в данных — дельты скрываются (`delta=None`),
KPI остаются с абсолютными значениями.

## Подписи на главной

**Под заголовком** (между title и KPI):

> _За период: 01.02.2026 – 30.04.2026 · 3 мес · «Последние 3 месяца»_

**Над «Анализом запасов»:**

> ℹ️ _Состояние склада на конец Apr 2026 — независимо от фильтра периода._

## Edge cases

| Ситуация | Поведение |
|---|---|
| Пустой период (нет данных в окне) | `st.warning(...)`, графики не рисуются, карточки запасов отображаются |
| Кастом: «От» > «До» | `st.error(...)`, графики не рисуются |
| Кастом: даты вне диапазона данных | Не падаем, см. «пустой период» |
| Предыдущего окна нет (например, «Весь период») | Дельты KPI скрыты |
| Меньше данных, чем пресет (например, «Последние 12» при 8 мес) | Окно clamps до доступной истории |

## Session state

- `period_preset` — выбор пресета
- `custom_from`, `custom_to` — даты при «Кастом»
- `granularity` — текущая (Неделя / Месяц)

Все ключи инициализируются при первом запуске. Смена пресета триггерит
пересчёт `granularity`.

## Тестирование

1. **Визуально:** каждый пресет → KPI меняются, графики меняются, карточки запасов остаются как были.
2. **Python-проверка:** для каждого пресета посчитать ожидаемые KPI через pandas вручную, сравнить с дашбордом (по [feedback_dashboard_verification]).
3. **Sanity-check:** для «Весь период» KPI = текущие значения дашборда (9 960 / 8 425 / 1 342 / 2 627 352 ₽). Это регресс-тест.
4. **Edge cases:** кастом `1.1.2024 – 1.3.2024` → warning. Кастом «До < От» → error.
5. **Гранулярность:** смена на «Последний месяц» → auto-Неделя. Ручной клик «Месяц» → 1 точка (это OK для KPI-режима).

## Что НЕ делается в этом тикете

- Сравнение двух периодов side-by-side (отдельная задача).
- Сохранение выбранного периода между сессиями (требует cookies/URL-params).
- Снапшот склада на конец выбранного периода (Вариант C в брейнсторме — на будущее).
- ABC-анализ артикулов внутри периода.
