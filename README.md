# Ozon Аналитика — Спутник Ключи

[![Open in Streamlit](https://static.streamlit.io/badges/streamlit_badge_black_white.svg)](https://ozon-sputnik.streamlit.app/)

**Публичная ссылка:** https://ozon-sputnik.streamlit.app/ — обновляется автоматически при `git push origin main` (Streamlit Community Cloud деплоит из публичного зеркала [`VADIMBAZ/ozon-analytics`](https://github.com/VADIMBAZ/ozon-analytics)).

Streamlit-дашборд для анализа продаж и запасов товаров на Ozon (артикулы `*SK2024` — ключи от умного домофона Спутник).

Объединяет два источника данных Ozon:

- **Заказы** (`sales results …`) — построчные заказы со статусами, для KPI и динамики продаж.
- **Аналитика** (`analytics_report_…`) — месячные агрегаты Ozon: заказано / доставлено / отменено / остаток на конец / дни без остатка. Используется для стокаут-логики и анализа запасов.

## Быстрый старт

```bash
./run.sh           # запуск в текущем терминале (с логами в stdout)
./run.sh start     # запуск в фоне на порту 8501 (логи: /tmp/streamlit.log)
./run.sh start 8502  # фон, кастомный порт
./run.sh status    # проверить, запущен ли
./run.sh stop      # остановить
```

При первом запуске скрипт сам создаёт `.venv` и ставит зависимости из `requirements.txt`.

Откройте http://127.0.0.1:8501 (или указанный порт).

### Ручной запуск

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
streamlit run app/main.py
```

Требуется Python ≥ 3.11 (тестировалось на 3.11 и 3.14).

## Источники данных

### 1. Заказы — `sales results …`

Перечислены явно в `app/services/loader.py:SALES_FILES`. Сейчас:

```python
SALES_FILES = [
    'sales results 01.09.2025-30.11.2025.numbers',
    'sales results 01.12.2025-28.02.2026.numbers',
    'sales results 01.03.2026-31.03.2026.numbers',
    'sales results 01.04.2026-30.04.2026.csv',
]
```

Формат: `.numbers` читается через `numbers-parser`, `.csv` — через `pandas.read_csv(sep=';')`.

Используемые колонки: `Принят в обработку` → `date`, `Статус`, `Артикул`, `Количество`, `Оплачено покупателем`.

### 2. Аналитика — `analytics_report_…`

Подхватываются автоматически по glob `analytics_report_*.xlsx`.

**Канонический формат имени:** `analytics_report_<Месяц> <Год>.xlsx`
(например, `analytics_report_Апрель 2026.xlsx`).

**Регрессионная защита:** loader пропускает любые файлы с подчёркиванием во «внутренней» части имени — это бэкапы (`*_полный.xlsx`, `*_до26.xlsx`, `2026-03-26_12_44.xlsx`).
А если на (артикул, период) находится больше одной строки — кидается `ValueError` с указанием дублей, чтобы анализ не считал стокауты по испорченным данным.

Структура xlsx (хардкод-смещения в loader): метаданные периода в `iloc[0,0]`, данные с 13-й строки, ключевые колонки — `8` (Артикул), `17` (Заказано), `19` (Доставлено), `23` (Отменено), `27` (X из Y дней без остатка), `28` (остаток на конец, `–` означает физическое отсутствие). `NaN`-остаток = «товар физически закончился» (не «нет данных») — критично для расчёта стокаутов.

### 3. Юнит-экономика — `Юнит-экономика_…`

Помесячные финансовые выгрузки `Юнит-экономика_DD.MM.YYYY-DD.MM.YYYY.xlsx` лежат в `data/unit_economics/`. Один файл = один календарный месяц (1-е по последнее число); произвольные диапазоны пропускаются с warning. Парсер: `load_unit_economics()` в `loader.py`.

Что забираем (25 финансовых полей): Цена, Заказано/Доставлено/Возвращено, Выручка, Баллы за скидки, Программы партнёров, Вознаграждение Ozon, Эквайринг, Логистика (+ ПВЗ, обработка отправления, размещение), Возвраты (обработка + обратная логистика), Реклама (клик + заказ), Доля от продаж, Прибыль за шт, Прибыль за период.

**Себестоимость в файле = 0** (Озон её не знает) — реальная COGS задаётся виджетом в сайдбаре, по умолчанию **25 ₽/шт**.

### Игнорируемые артикулы

`SKIP_ARTICLES = {'77SK2024T'}` — служебный/тестовый, исключается из обоих источников.

## Что показывает дашборд

### Сайдбар

**📅 Период анализа** (сверху сайдбара):
- Selectbox с пресетами: **Весь период · Последний полный месяц · Последние 3 / 6 / 12 месяцев · Кастом**.
- Дефолт при первой загрузке: **«Последние 3 месяца»**.
- При выборе **Кастом** появляются два `date_input` («От» / «До»).
- Под селектбоксом — подпись `01.02.2026 – 30.04.2026 · 3 мес` (длительность окна).
- **На что влияет:** KPI, инсайты, графики (динамика, упущенная выручка, остатки+приходы, % отмен), сводная таблица, **юнит-экономика**.
- **На что НЕ влияет:** «Анализ запасов» — это всегда состояние склада на конец последнего месяца аналитики. Над сводкой запасов явная подпись `ℹ️ Состояние склада на конец <месяц> — независимо от фильтра периода`.

**💰 Себестоимость** (после фильтра периода):
- Одна цифра на все артикулы, дефолт **25 ₽/шт**.
- Меняется через `number_input` в сайдбаре → пересчитываются **оба** раздела: «Заморожено по себестоимости» (карточки запасов) и «Прибыль чистая / Маржа %» (юнит-экономика).
- Хранится в `st.session_state['cost_per_unit']` (в рамках сессии).

**📦 Артикулы:**
- Чекбоксы с миниатюрами (`ARTICLE_IMAGES`: Ozon-CDN или base64 из `app/images/`).
- Кнопки `✅ Все` / `✖ Сброс`.

**Гранулярность временного ряда: Неделя / Месяц.**
Автоматически переключается при смене пресета (≤ 31 дня → Неделя, > 31 → Месяц). Ручной выбор сохраняется до следующей смены пресета.

Артикулы отсортированы по объёму доставок (сначала топ).

### KPI-ряд (4 метрики)

Все четыре считаются по **выбранному периоду** (из фильтра в сайдбаре).

| Метрика | Источник | Дельта |
|---|---|---|
| Всего заказано (шт) | сумма `Количество` по выбранным | vs **предыдущее окно такой же длины** |
| Доставлено (шт) | строки со `Статус == 'Доставлен'` | vs предыдущее окно |
| Отменено (шт) | строки со `Статус == 'Отменён'` | inverse (рост = плохо) |
| Выручка (доставл.) | сумма `Оплачено покупателем` по доставленным | vs предыдущее окно |

Если предыдущее окно такой же длины не помещается в исторические данные — дельты скрываются.

### Аналитические наблюдения (`build_insights`)

Авто-инсайты, развёрнуты по умолчанию:

- 🏆 **Лидер продаж** — артикул №1 по доставкам, доля, отрыв от №2.
- 💰 **Средний чек** — самый высокий и низкий по артикулам.
- 🔴 **Высокий % отмен** — если у артикула отмен > 1.3 × среднего и > 10%.
- ⚠️ **Стокаут съел продажи** — артикулы, где остаток < средних продаж и факт < 50% средних.
- 📈 **Резкий рост** — последний месяц вырос ≥ 1.8× к предыдущему.
- 📊/📉 **Общий тренд** — среднее трёх первых vs трёх последних месяцев.
- 🚨 **Нулевой остаток** — где `остаток_конец == 0` в последнем месяце.

### Динамика продаж

Plotly-line chart, гранулярность Неделя/Месяц:

- Тонкие линии — по каждому выбранному артикулу.
- Жирная линия **Итого** (если выбрано ≥ 2) — сумма всех выбранных.

### 💸 Упущенная выручка из-за стокаутов

Stacked bar по месяцам + сумма по артикулу.

**Формула:**
1. Берём «нормальные» месяцы — где `остаток_конец ≥ Доставлено` (товар не заканчивался). База считается **по всей истории артикула**, а не по выбранному периоду — иначе при фильтре «Последние 3 месяца» у артикулов с хроническим дефицитом или коротким окном вся нормальная база отрезается, avg = 0 и потери исчезают.
2. `avg_monthly_sales` = среднее по нормальным месяцам всей истории.
3. Для каждого месяца **внутри выбранного периода**, где `остаток_конец < avg_monthly_sales` **или** остаток = `NaN` (товар физически отсутствует):
   `упущено = (avg_monthly_sales − Доставлено) × средний_чек`.

**Важно:** `NaN`-остаток (`–` в xlsx) трактуется как стокаут, а не как «нет данных» — иначе теряются полные стокауты вроде `06SK2024` в апреле 2026.

**Известные ограничения формулы** (есть план их адресовать — обсуждается отдельно):

- «Двойной счёт»: один и тот же месяц может попасть и в «нормальные» (для базы среднего), и в «стокауты» (если факт. продажи ниже среднего). Это занижает базу или завышает потери в одном и том же месяце.
- Артикулы с **хроническим дефицитом** (нет ни одного нормального месяца) дают `0` потерь, хотя на практике именно по ним потери максимальны.
- Поле `дней_без_остатка` из Ozon не используется. Попытка перейти на него (2026-05-12) была откачена: смысл поля у Ozon — не «дни с нулевым остатком на складе», а нечто другое (медиана 20.5 из 30 даже у активно продающихся артикулов). Без понимания точной семантики поля строить на нём формулу нельзя.
- Нет сезонности, тренда, разделения цен по месяцам, кросс-эластичности между артикулами.

Под графиком — итог за весь период и топ-3 артикулов по потерям.

### Остаток на конец месяца и приходы

Двухпанельный subplot:

- **Сверху** — столбцы остатков по месяцам, по всем артикулам.
- **Снизу** — расчётные приходы:

  `приход = остаток_конец − остаток_предыдущего_месяца + Доставлено`

  Порог `INCOMING_THRESHOLD = 20` шт — мельче считаем шумом (возвраты, корректировки Ozon), а не реальной поставкой.

### Анализ запасов (карточки)

**Сводка наверху** (4 числа):

- Всего на складе (шт).
- Розничная цена замороженного — `остаток × средняя_цена_продажи`.
- Себестоимость — `остаток × cost_per_unit` (дефолт **25 ₽/шт**, регулируется в сайдбаре в разделе «💰 Себестоимость»).
- Наценка — разница розницы и себестоимости.

**Инвариант:** если `остаток_конец = NaN` (в xlsx стоит `–` — товара физически нет на складе), карточка **строится** с `stock = 0` и попадает в статус «Стокаут». Это тот же инвариант, что в расчёте упущенной выручки: NaN не означает «нет данных», означает «нет товара». Без этого правила лидер продаж с пустым складом исчезает из «Анализа запасов» (исторический баг с `06SK2024` в Apr 2026).

**Карточки на 3 в ряд**, отсортированы по срочности действий:

1. **Стокаут** (запас < 0.5 мес) — красный, требует срочной поставки.
2. **Заканчивается** (0.5–1 мес) — оранжевый, пора заказывать.
3. **Перезатарка** (> 6 мес) — красный, слишком много заморожено.
4. **Избыток** (3–6 мес) — жёлтый.
5. **Норма** (1–3 мес) — зелёный.

Внутри группы — по объёму замороженных денег по убыванию.
`запас_мес = остаток / средние_продажи_в_мес`.

**Почему 2 «красных» статуса:** стокаут (< 0.5 мес) и перезатарка (> 6 мес) — разные проблемы. Стокаут блокирует продажи и ставится наверх; перезатарка только замораживает деньги.

### 💰 Юнит-экономика

Финансовая раскладка по данным Озон + наша себестоимость. Реагирует на оба фильтра (период + артикулы). Делится на 3 таба.

**Сверху раздела:**
- KPI: Выручка / Прибыль до COGS / Прибыль чистая / Маржа чистая % (с дельтами vs предыдущий период такой же длины).
- Сводка структуры расходов в % выручки: Комиссия Озон / Логистика / Возвраты / Реклама / Себестоимость.

**Формулы:**

```
Прибыль до COGS = из файла Озон (Прибыль за период)
COGS итого = cost_per_unit × (Доставлено − Возвращено)
Прибыль чистая = Прибыль до COGS − COGS итого
Маржа чистая % = Прибыль чистая / Выручка × 100
```

**Таб 1 — 📈 Динамика маржи:**
- Линии: Маржа % по месяцам (агрегат жирной линией + по каждому SKU тонкими пунктирами, цвета из общей палитры артикулов).
- Стек-бар: Прибыль чистая ₽ по месяцам (стек по SKU).
- Heatmap: SKU × месяц, цвет = Маржа %.

**Таб 2 — 💸 Структура расходов:**
- Waterfall: Выручка → +Компенсации (баллы + программы партнёров) → −Комиссия Озон → −Логистика → −Возвраты → −Реклама → −Себестоимость → Прибыль чистая.
- Селектор «Все выбранные / по конкретному SKU».
- Стек-бар 100% — структура расходов в % выручки помесячно (рисуется при ≥ 2 месяцах данных).

**Таб 3 — 🎯 Упущенная прибыль:**

```
Упущенная прибыль = Упущенная выручка × Маржа чистая % этого SKU
```

Таблица `SKU | Упущенная выручка | Маржа % | Упущенная прибыль` (отсортирована по упущенной прибыли, с подсветкой градиентом). Переводит «упущено N руб. выручки» в «упущено M руб. реальной прибыли».

**Edge cases:**
- Нет unit-файла за месяц из analytics → пропуск месяца, без падения.
- Файл с произвольным диапазоном (не календарный месяц) → skip + warning.
- Пустой период (нет unit-данных в окне) → warning, KPI/табы не рисуются.
- COGS > Цена → отрицательная маржа показывается как есть.

## Как добавить новый месяц данных

### Через UI дашборда (рекомендуемый путь)

В сайдбаре раздел **🔄 Обновить данные** (доступен на https://ozon-sputnik.streamlit.app/ при настроенных секретах — см. [Настройка деплоя](#настройка-деплоя)):

1. Скачай из Seller-кабинета Ozon три файла за прошлый месяц:
   - `analytics_report_<Месяц> <Год>.xlsx` (раздел «Аналитика → Графики»);
   - `sales results <DD.MM>-<DD.MM>.<numbers|csv|xlsx>` (раздел «Заказы → FBO/FBS»);
   - `Юнит-экономика_DD.MM.YYYY-DD.MM.YYYY.xlsx` (раздел «Аналитика → Юнит-экономика», выбрать календарный месяц 1-е по последнее).
2. Открой дашборд, разверни в сайдбаре «🔄 Обновить данные», введи пароль.
3. Перетащи все три файла в поле загрузки → кнопка **«Применить»** (поддерживается мульти-загрузка).
4. Дашборд сам:
   - валидирует структуру (если файл не тот — ошибка, ничего не коммитится);
   - переименовывает в каноническое имя (даже если ты переименовал по-своему);
   - коммитит через GitHub API в публичный mirror;
   - старая версия файла сохраняется в **git history** (откат: `git show <hash>:<filename>`).
5. Через ~90 сек Streamlit Cloud пересоберёт дашборд → обнови страницу.

Все валидации делаются до коммита, поэтому **либо все файлы загружены чисто, либо ничего**.

### Через файловую систему вручную (запасной путь)

Если UI-загрузчик недоступен или хочется работать локально:

1. **Заказы:** положить новый файл в корень проекта (`sales results <DD.MM.YYYY>-<DD.MM.YYYY>.<ext>`). Loader подхватит автоматически по glob-маске.
2. **Аналитика:** сохранить отчёт Ozon как `analytics_report_<Месяц> <Год>.xlsx` (русский месяц + пробел + год, **без подчёркиваний внутри**). Положить в корень. Loader подхватит автоматически.
3. **Юнит-экономика:** положить в `data/unit_economics/Юнит-экономика_DD.MM.YYYY-DD.MM.YYYY.xlsx` (только календарные месяцы). Loader подхватит автоматически; некалендарные диапазоны skip + warning.
4. Бэкапы хранить с подчёркиванием в имени (`analytics_report_Март 2026_полный.xlsx`) — они отфильтруются.
5. Перезапустить Streamlit (`./run.sh stop && ./run.sh start`), либо нажать "Rerun" в UI.

Кэширование: `@st.cache_data` на `get_data()` — после изменения файлов нужен hard reload или перезапуск.

### Публичный URL и автодеплой

- **Прод-ссылка:** https://ozon-sputnik.streamlit.app/
- **Деплоится из:** [`VADIMBAZ/ozon-analytics`](https://github.com/VADIMBAZ/ozon-analytics) (публичный mirror).

- **Push идёт в оба repo одной командой** `git push origin main` (см. `.git/config` — для `origin` настроены два `pushurl`).

### Включение UI-загрузчика (одноразово)

Без этих секретов блок «🔄 Обновить данные» отображается в read-only режиме и ничего не делает.

1. **Создать fine-grained PAT:**
   https://github.com/settings/personal-access-tokens/new
   - Resource owner: **VADIMBAZ**
   - Repository access: **Only select repositories** → `VADIMBAZ/ozon-analytics`
   - Repository permissions:
     - **Contents: Read and write**
     - **Metadata: Read-only** (выдаётся автоматически)
   - Expiration: 90 дней (безопасно) или 1 год (удобно).

2. **Положить секреты в Streamlit Cloud:**
   Открыть https://share.streamlit.io → app **ozon-sputnik** → **Settings** → **Secrets** → вставить:
   ```toml
   GITHUB_TOKEN = "github_pat_xxx..."
   UPLOAD_PASSWORD = "любая_строка"
   ```
   Сохранить. Streamlit сам перезагрузит контейнер.

3. Открыть https://ozon-sputnik.streamlit.app/ → в сайдбаре проверить, что в «🔄 Обновить данные» появилась форма с полем пароля и file_uploader'ом.
